Summary


Purpose: Generate Java~JRecord code to Read/Write Cobol Data files using a Cobol Copybook in java.
License: GPL
Queries: Use the JRecord Forums.
Problems: Use either the JRecord Forums or JRecord Tickets.

Future: Future development will be based on the level of interest / feedback. So if you find this program is useful to you, please Let me know !!!, provide feedback and get involved !!! also documentation would be useful.

Changes in this release include:

  • Adding Java interface.
  • Adding file Analysis to work out file attributes from a sample file.
  • Adding Schema Walker to allow user to generate there own code using just java.

CodeGen Template interface

Java Interface

There is a builder style interface to CodeGen. In the following example, the analyseDataFileSetAttributes codegen to try and determine file attributes from the sample data file. 

 5         CodeGenInterface.TEMPLATES.newTempateBuilder("DTAR020.cbl")
 6                 .setDataFile("DTAR020.bin")
 7                 .analyseDataFileSetAttributes()
 8                 .setTemplate(StandardTemplates.STANDARD)
 9                 .setPackageName("dtar020.standard")
10                 .setOutputDirectory("G:\\Temp\\Gen\\TestCodeGen\\dtar020")
11                 .generateJava()
12                 .writeTemplateBuilderCode("build");
In this example, CodeGen will try and workout the File Attributes from a sample data file (methods setDataFile and analyseDataFileSetAttributes). The main Template methods are:
  • setDataFile method defines a data file to be analyzed by CodeGen. The analyseDataFileSetAttributes and createRecordDecider
  • analyseDataFileSetAttributes method tells CodeGen to analyze the sample file (from setDataFile method) and work out the file encoding and file-organisation.
  • createRecordDecider method tells CodeGen to analyse the sample file (from setDataFile method) and create a Record-Decider. This only makes sense for a multi-record file.
  • setTemplate - define the template to be generated:
    • StandardTemplates.STANDARD - creates a basic Java/JRecord programs to read / write the file.
    • StandardTemplates.POJO - creates a java pojo and code to convert it to/from Cobol-Data. The generated POJO has no reference JRecord.
    • StandardTemplates.POJO_INTERFACE - same as the POJO template but with the addition of Interfaces
    • StandardTemplates.LINE_WRAPPER - create a class to wrap a JRecord line and proved get/set methods for the fields.
  • setFileOrganisation Set the file organization or structure, This option is also set by the setDataFile/analyseDataFileSetAttributes/. Options include:
    • FileOrganisation.TEXT - Standard Windows/Linux text file
    • FileOrganisation.FIXED_LENGTH - All records (lines) have a standard Fixed length. There is no line seperator between records.
    • FileOrganisation.MAINFRAME_VB - mainframe VB file
  • setFont - set the encoding/font
  • setCopybookSplitOption Tell CodeGen if/how to split a Cobol-Copybook into multiple records. Options are:
    • CopybookSplit.SPLIT_NONE - Single record file, the copybook represents one record.
    • CopybookSplit.SPLIT_01_LEVEL - split the copybook into seperate records via the 01 level (a record for each 01 level)
    • CopybookSplit.SPLIT_REDEFINE - split the copybook on a redefine
  • setRenameDuplicateFields - rename fields to ensure no duplicates.
  • setPackageName - define the package name of the generated code.
  • setOutputDirectory - set the directory where the generated code is to be written to.
  • generateJava - Generate Java Code.

Renaming duplicate fields

JRecord `flattens` Cobol groups to create a single/multiple records. This can lead to duplicate field-names. The setRenameDuplicateFields(true) tells JRecord to rename duplicate fields to ensure all fields are unique.

30         CodeGenInterface.TEMPLATES.newTempateBuilder(iob)
31                 .setRenameDuplicateFields(true)
32                 .setPackageName("tstr.pojo.rename")

Running Java from the shell

As of java 11 you can compile and run java programs from the command line. See CodeGenExampleJava/template for examples.

    ./runJavaCodeGen.sh DTAR020.cbl

In Windows you can do: 

    runJavaCodeGen DTAR020.cbl

User written Schema Walker

The Schema walker was provided to allow users to write there own Code Generation code in Java.

The schema walker walks the records and fields of a schema calls a user supplied listener class to process each record/field. This listener supplied class can generate code amongst other things.

The following walker example will list the records and fields in a Cobol copybook:

 3         ICobolIOBuilder ioBuilder = JRecordInterface1.COBOL
 4                 .newIOBuilder(TstIoBldrGen01.class.getResource("DTAR020.cbl").getFile())
 5                         .setFileOrganization(IFileStructureConstants.IO_FIXED_LENGTH)
 6                         .setFont("cp037");
 7         
 8         CodeGenInterface.WALKER.newSchemaWalker(ioBuilder)
 9             .walk(new SchemaWalkerAdapter() {
10 
11                 @Override public void startRecord(LayoutDef schema, RecordDef record) {
12                     System.out.println("\n\t" + record.getCobolName());
13                 }
14 
15                 String spaces = "                                  ";
16                 @Override public void processField(LayoutDef schema, RecordDef record, FieldDef fieldDtls) {
17                     Usage usage = fieldDtls.getCobolItem().getUsage();
18                     System.out.println("\t\t" 
19                             + (fieldDtls.getCobolName() + spaces).subSequence(0, 32)
20                             + fieldDtls.getPos() + "\t" + fieldDtls.getLen() + "\t" + fieldDtls.getDecimal()
21                             + "\t" + fieldDtls.getCobolItem().getPicture()
22                             + "\t" + (usage == Usage.NONE ? "" : usage)
23                             );
24                 }
25             });

CodeGen Templates

The CodeGen program reads a Cobol-Copybook and Generates Java~JRecord Code to read/write the Cobol-Data file. CodeGen is built around Templates. A CodeGen Template is a directory of Velocity Templates with a Generate.properties control file. Each CodeGen Template generates one or more programs/files.

The templates I suggest looking at are:

  • pojo - will convert the Cobol Records to / from Java objects. It can have problems if there are duplicate field names, redefines or complicated Cobol files
  • lineWrapper - This creates a Java Wrapper around a JRecord~Line. It is good when only accessing a few fields or the files is to complicated for the pojo template
  • standard generates basic low-level JRecord code without any java representation. This template will handle Cobol Copybooks the other templates will not.

CodeGen code-templates include:

  • basic - generate very basic JRecord Read/Write example code.
  • standard - like the basic template only a Class holding Cobol Field Names
    • is generated and used. The initial release of CodeGen focuses on the Standard Template. There are examples in the CodeGenExample/standard directory
  • lineWrapper - Is an extension on the standard template. It generates a line-wrapper the has getter/setters for the Cobol fields.
  • lineWrapperPojo - Is an extension on the lineWrapper template. It generates an interface for the line-wrapper + an alternative Pojo implementation of the line.
  • pojo - Provides basic Pojo (plain old java objects) with readers / writers for the Pojo
  • schemaClass An extension of standard template. It generates a full schema class. The Schema details can be either loaded from a Cobol copybook or Created at run time. There are likely to be problems with Occurs Depending. See CodeGenExample/otherTemplates for examples of generating schemaClass templates.
  • javaPojo Generates a Pojo / Javabean class + reader, writer and schema classes. This template currently supports single record files with no redefines (and no utf-8/16 files). See CodeGenExample/otherTemplates for examples of generating javaPojo templates.

Standard CodeGen Template

The standard is a fairly simple "Template", it generates 3 files
  1. A field name Class. This field name class holds all the Cobol Field names as java final variables. Rather than having to remember the Cobol Field names, you can then use Auto-Completion in your IDE. There will be issues if there are duplicate field names.
  2. A sample Read program.
  3. A sample Write program.

Line Wrapper Template

The lineWrapper is an extension of the standard "Template", it adds a LineWrapper class
  1. The LineWrapper class - this class has getter/setters methods for the Cobol Fields. These getter/setter call the getFieldValue(...) method of the "wrapped" line.
  2. A field name Class. This field name class holds all the Cobol Field names as java final variables. Rather than having to remember the Cobol Field names, you can then use Auto-Completion in your IDE. There will be issues if there are duplicate field names.
  3. A sample Read program.
  4. A sample Write program.

Example of Using CodeGen generated classes

The LayoutCompare sub-project uses java class's generated from a Cobol Copybook JR_Schema_Test.cbl using the lineWrapper CodeGen Template.

The project uses the following classes are generated by CodeGen:

  • FieldNamesJrSchemaTest.java - Holds the Cobol Field names. It used by the other classes to access the Fields
  • LineCopybookRecordJR.java - Line-Wrapper class for the Copybook Record in the above Cobol Copybook.
  • LineDialectRecordJR.java - Line-Wrapper class for the Dialect Record in the above Cobol Copybook.
  • LineSchemaRecordRecordJR.java - Line-Wrapper class for the Schema-Record Record in the above Cobol Copybook.
  • LineFieldRecordJR.java - Line-Wrapper class for the Field Record in the above Cobol Copybook.

These generated classes are organised into a Layout - Record - Field Hierachy by the LayoutDtls and RecordDtls classes: 

       LayoutDtls
            |      1 to many
            +------------------- RecordDtls
                                    |        1 to many
                                    +---------------------- LineFieldRecordJR

Finally there are 2 IO classes:

  • The WriteJrSchema.java class will write JRecord-Layout to an external file (using the above generated classes).
  • The ReadJrSchema.java class will read the file into a LayoutDtls classes. These LayoutDtls are compared by program CompareCopybookDetails.java with a JRecord-Layout that was generated with current version of JRecord.

The LayoutCompare sub-project is in the CodeGenExample/ExampleOfUsingCodeGen/ directory.

What is a CodeGen Template

A CodeGen Template is a Directory holding a Generate.properties file. The Generate.properties file holds

  • Velocity Templates to be Generated
  • The output names of the Generated Velocity templates.
  • Various options related to the Template

A basic Template looks like: 

1 skel.0=2
2 ##
3 skel.1.template=&template./exampleRead.vm
4 skel.1.output=CodeGenExample/Read&suffix..java
5 skel.2.template=&template./exampleWrite.vm
6 skel.2.output=CodeGenExample/Write&suffix..java
7 ##
8 requirePackageId=N

Options include:

  • The skel.0=2 tells CodeGen there are 2 Velocity Templates to be Generated
  • The skel.1.template=&template./exampleRead.vm to generate the exampleRead.vm file in the current directory.
  • The skel.1.output=CodeGenExample/Read&suffix..java tells CodeGen the output name. &suffix. is replaced by the Schema (or layout) name.
  • The requirePackageId=N tells CodeGen a java package-id is not required.
  • The splitAllowed=Y tells CodeGen the split option can be used (i.e. multi-record-type files are allowed).

As well as generating Velocity Template for the Schema (Cobol Copybook), you can generate a Velocity Template for each record in the copybook by adding a skel.index.genAt=record: 

25 skel.3.template=lineWrapperPojo/DataJR.vm
26 skel.3.output=&directory./data/Line&recordSuffix.JR.java
27 skel.3.genAt=record
28 skel.3.description=A "line wrapper" class that provides get.../set... methods for the Cobol fields

Note: &recordSuffix. is replaced by the record name.

Velocity Template

To learn about writing Velocity Templates, you need to look through the existing Templates or ask me.

Velocity skeltons get 2 variable (schema / layout) level or 3 variables (record level).

variableDescription
${packageId}Holds the java package-id of the class. i.e 
1 package ${packageId};
${generateOptions}This variable implement the IGenerateOptions interface. For example ${generateOptions.schemaDefinition} retrieves the schema definition.

To loop through all the record-Types in a schema: 

19 #*  *##foreach( $r in ${generateOptions.schemaDefinition.getRecords()}) 
20     public static final Record${r.extensionName} RECORD_${r.constantName} = new Record${r.extensionName}();
21 #*  *##end

To loop through all the fields in every record Record: 

23 #*  *##foreach( $r in ${generateOptions.schemaDefinition.getRecords()}) 
24     public static class Record${r.extensionName} {
25 #*    *##foreach($f in ${r.getFields()})        
26 #*      *##if (! ${f.isArrayItem()})
27        public final String ${f.javaName} = "${f.fieldDetail.getLookupName()}";
28 #*      *##end
29 #*    *##end

${currentRecord}Record templates also get the record being generated (${currentRecord}). Usage: 
10 #*  *##foreach($f in ${currentRecord.getFields()}) 
11 #*      *##if (! ${f.isArrayItem()})    
12     public ${f.getJavaType()} get${f.extensionName}();
13 #*      *##end
14 #*  *##end

For all the schema details (LayoutDef, RecordDef, FieldDef) extend JavaDetails They have the following methods available

getCobolName() or cobolNameGet the cobol name e.g. "My-Field-Name"
getConstantName() or constantNameGet the cobol name e.g. "MY_FIELD_NAME"
getClassName() or classNamevalid Class name e.g. "MyFieldName"
getExtensionName() or extensionNameCamel-Case name e.g. "MyFieldName"
getJavaName() or javaNameJava Camel-Case name e.g. "myFieldName"
